# 进校系统 — 学生变动场景功能设计与学情数据处理

生成日期：2026-06-09
讨论背景：已售卖的智慧校园产品，针对换班、休学、升年级、升学段、走班、行政班相关场景的优化迭代

---

## 一、关键决策问题与回答

### Q1: 产品目标定位
**问题：** 做这个进校系统的目标是什么？
**回答：** 已经售卖的智慧校园产品，进行目标场景下的优化迭代。

### Q2: 核心痛点
**问题：** 当学生换班、休学、升年级的时候，系统里具体会发生什么问题？
**回答：** 数据归属混乱。具体表现：
- 现在没有专门的换班操作功能，需要手动把学生从 A 班名单里删除再添加到 B 班
- A 班里的历史学情数据无法合理调整
- A 班老师、年级主任查看 A 班历史数据时，该学生的数据仍应保留
- 但查看详细任务和学生个人学情时，是否应包含历史数据，怎么处理有变动的学生，目前数据口径混乱不清

### Q3: 走班场景普遍性
**问题：** 学生"同时属于多个班"的情况在客户中普遍吗？
**回答：** 普遍存在。

### Q4: 数据源
**问题：** 学生数据源头是哪里？
**回答：** 自己维护（非从第三方教务系统同步）。

### Q5: 升年级模式
**问题：** 升级时是班级整体升级还是会打散重组？
**回答：** 两种情况都有。有的学校整班升，有的会打散重组。

### Q6: 走班复杂度
**问题：** 行政班/教学班的分类是否能覆盖所有场景？
**回答：** 更复杂。同一个学科，也可能存在固定的行政班教学 + 部分学生重组形成"专项竞赛班"。但有的学校所有学科都在同一行政班教学，没有走班概念。

### Q7: 系统现状
**问题：** 系统目前的班级类型支持情况？
**回答：** 目前系统已经支持创建班级时选择类型是"走班"还是"行政班"。

### Q8: 架构方案选择
**问题：** 三个方案（A-最小可行 / B-理想架构 / C-渐进方案）倾向哪个？
**回答：** 选 B — 理想架构（一次性做到位）。

## 二、确认的设计前提

1. **"班级"至少是两种东西：行政班（人的归属）和走班（课的归属）。** 系统已有此区分。
2. **学生-班级关系必须带时间维度。** 不是"张三属于 3 班"，而是"张三从 2025-09-01 到 2026-01-15 属于 3 班"。
3. **数据口径问题本质上是产品决策。** "A 班历史统计是否包含已转走的学生"需要明确的默认规则。
4. **"升年级"有两种模式：** A）班级整体升级（班不散人不变）；B）打散重组（旧班消亡新班诞生）。系统需同时支持。
5. **休学学生进入冻结状态。** 门禁权限立即撤销，学情数据完整保留，复学后可接续。

## 三、第二视角独立分析（Claude Subagent）

### 3.1 Steelman — 最强版本的表述

你们在做的是一个**时间维度的学生-组织关系图**。每条关系都是有时间边界的边，任何报表都可以按时间点或时间范围查询（"3 月 15 日那天 3 班有谁？" vs "整个学期贡献了 3 班聚合数据的都有谁？"）。

这不是一个带软删除的 CRUD 应用。它是一个**事件溯源的学籍台账**，数据源是"变更事件的序列"，"当前状态"只是最新的投影（projection）。做对了，就从根本上消灭了"数据孤儿"类 bug，让归属口径变成查询时的参数而非数据迁移问题。

### 3.2 最有揭示性的发现

> "升年级有两种模式：整班升和打散重组"

这揭示了系统真正的建模难题：**"班级"不是一个稳定的容器。** 它有时是一个穿越时间的持久实体（整班从高一升到高二），有时是一个只存在一学年的临时分组（分班后旧班不复存在）。

大多数团队会选一种模型然后发现另一种场景处理不了。两种模式共存意味着：**班级实体本身需要独立的生命周期**（创建→活跃→解散→归档），不依赖于里面的学生。

关键推论："班级解散"不能建模为"45 个学生的批量换班"，而必须是班级实体上的一等事件，学生重新分配是该事件的下游结果。

### 3.3 对前提的质疑

**质疑前提 3："数据归属规则是产品决策，不同学校可能有不同诉求"**

第二视角认为这可能是错的：
- 可配置的归属规则会让查询复杂度爆炸
- 每个聚合报表都需要"策略"参数，缓存失效要考虑多种可能答案
- 每个学校管理员还得理解一个他们从没要求过的抽象

**预测：** 去问 10 所学校同一个场景问题（"张同学 10 月 15 日转走了，A 班的学期考勤率是否包含她 9 月份的数据？"），8 所以上会给相同答案：
- 班级聚合统计 → 保留历史学生数据
- 学生个人档案 → 按时间线完整展示

**建议：** 先做 5 所学校场景访谈验证。如果收敛度 >80%，就硬编码一个规则，不做配置化。有付费客户明确要求不同规则时再开放配置。

### 3.4 如果只有 48 小时做原型

**技术选型：** PostgreSQL（时间范围表 `tstzrange` + 排他约束），Go 或 Node 后端，不做前端 — 只做 API + 种子数据 + Jupyter Notebook 演示查询。

**核心 3 张表：**

```sql
-- 班级表
class (
  id, type ENUM(administrative/teaching), name,
  grade_level, academic_year, status,
  valid_range tstzrange
)

-- 学生-班级关联表（带时间维度）
enrollment (
  id, student_id, class_id, role,
  valid_range tstzrange, event_id FK
)

-- 变更事件表
enrollment_event (
  id, type ENUM(transfer/promotion/leave/return/dissolution),
  payload JSONB, occurred_at, operator_id
)
```

**验证的 4 个操作：**
1. 换班（写事件 + 关闭旧关联时间范围 + 开启新关联）
2. 整班升级（更新 class.grade_level，关联不动）
3. 打散重组（解散事件 + 批量新关联）
4. 休学/复学（冻结关联 + 新事件恢复）

**验证的 2 个查询：**
- "给我 3 班在 X 日期的花名册"（`WHERE valid_range @> X::timestamptz`）
- "3 班 9 月-12 月的考勤人数，包含该时间段内所有曾在班学生"

**跳过：** 走班、UI、权限、实际学情数据接入、口径可配置化。原型的目的是验证"时间关联 + 事件记录"能让"谁在哪里在什么时候"这个问题变得可查询。

## 四、综合结论

### 4.1 两个视角一致的地方

| 结论 | 说明 |
|------|------|
| 事件溯源 + 时间维度是正确方向 | 不直接修改关联，而是记录"发生了什么"，查询时按事件流重建状态 |
| 班级需要独立生命周期 | 班级不只是学生的容器，它自己有创建/活跃/解散/归档的状态 |
| "升年级 ≠ 换班"必须区别建模 | 整班升级是班级属性变化，打散重组是班级生命周期结束+新班诞生 |
| 先硬编码归属规则，不过早做配置化 | 验证大多数学校的诉求一致后，固定一个规则 |

### 4.2 推荐的默认归属规则（待验证）

- **班级维度聚合统计（如"A 班 3 月考勤率"）：** 包含当时所有在班学生（含后来转走的），因为他们确实贡献了那个月的数据
- **学生个人维度（如"张三的学情档案"）：** 按时间线完整展示，跨班跨年级无断裂，清晰标注"高一(3)班期间"/"高二(5)班期间"
- **班级当前名单：** 只显示当前在班学生（转走和休学的不显示）

### 4.3 设计简化原则

保持现有的班级双类型（行政班 / 走班）不变，不引入新的班级类型概念。"专项竞赛班"在系统中就是"走班"的一种使用方式。核心改动聚焦在底层数据关联和操作流程上。

## 五、后续产品设计思路

### 5.1 第一优先级 — 数据模型升级

**改动：** 学生-班级关联表增加 `start_date` / `end_date` / `event_id`

**影响：**
- 所有查询学情数据的地方，需要加入时间过滤条件
- 历史数据迁移：现有关联补上 `start_date`（取学生入学或班级创建时间）

### 5.2 第二优先级 — 管理操作入口

需要新增的操作页面：

| 操作 | 触发者 | 系统行为 |
|------|--------|----------|
| 学生换班 | 教务老师 | 关闭旧关联（end_date=今天）+ 创建新关联（start_date=今天）+ 记录事件 |
| 学生休学 | 教务老师 | 学籍状态→休学 + 关闭所有班级关联 + 撤销门禁权限 + 记录事件 |
| 学生复学 | 教务老师 | 学籍状态→在读 + 创建新关联（可能换了班）+ 恢复门禁权限 + 记录事件 |
| 整班升年级 | 系统/教务 | 班级的 grade_level +1 + academic_year 更新，学生关联不变 |
| 打散重组 | 教务老师 | 旧班标记为"已解散" + 批量关闭旧关联 + 学生分配到新班 + 批量创建新关联 |
| 升学段 | 系统/教务 | 初中学籍→归档，高中学籍→新建（如果是完全中学一贯制） |

### 5.3 第三优先级 — 学情数据展示口径

**班级视角（老师/年级主任看班级数据）：**
- 默认展示"在查询时间范围内曾在班"的学生数据
- 当前花名册只显示"现在在班"的学生
- 已转走学生在历史统计中有标记（如灰色或带"已转出"标签）

**学生视角（查看个人学情）：**
- 时间线模式，按学期/班级分段展示
- 清晰标注每段归属的班级和时间
- 跨班无断裂，完整的个人成长档案

**老师视角（班主任/科任老师）：**
- 班主任看到的始终是行政班视角
- 走班科任老师看到的是教学班视角
- 两个视角独立，不混淆

### 5.4 学籍状态机

```
在读 (active)
  ├── → 休学 (suspended)  [可逆：复学回到在读]
  ├── → 转学 (transferred) [不可逆：离开本系统]
  └── → 毕业 (graduated)  [不可逆：学籍归档]

休学 (suspended)
  └── → 在读 (active)     [复学]
```

状态变更联动：
- 休学 → 自动撤销门禁权限，关闭所有班级关联
- 复学 → 恢复门禁权限，需手动分配班级
- 转学/毕业 → 永久撤销权限，数据归档保留

### 5.5 升年级的产品流程设计

**整班升级模式：**
1. 学期末，系统提示"新学年即将开始，是否执行升年级？"
2. 管理员确认后，系统批量更新所有班级的 grade_level
3. 毕业年级（高三/初三）的班级标记为"已毕业"，学生学籍→毕业
4. 学生-班级关联不变，无需逐个操作

**打散重组模式：**
1. 管理员创建新学年的班级结构
2. 系统提供"分班工具"，可以按规则或手动将旧班学生分配到新班
3. 确认后，旧班全部标记为"已解散"（end_date），新关联批量创建（start_date）
4. 旧班数据完整保留，可追溯

### 5.6 需要验证的产品假设

在开发前，建议对 5 所学校做场景访谈，验证：

1. "张同学 10 月 15 日从 A 班转到 B 班。A 班的学期考勤统计是否包含张同学 9 月份的数据？" → 预期答案：包含
2. "张同学休学后复学，复学后分到了不同的班。他的个人档案怎么展示？" → 预期答案：按时间线展示
3. "走班（教学班）的成绩数据，年级主任想看的时候按什么维度聚合？" → 需要确认
4. "升年级时打散重组，老师想回看去年带的班的数据，能看到吗？" → 预期答案：能，旧班数据归档但可查

## 六、技术方案参考（供研发团队参考）

### 核心表结构变更建议

```sql
-- 学生-班级关联表（核心改动：加时间维度）
ALTER TABLE student_class ADD COLUMN start_date DATE NOT NULL;
ALTER TABLE student_class ADD COLUMN end_date DATE;  -- NULL 表示当前有效
ALTER TABLE student_class ADD COLUMN event_id BIGINT; -- 关联到变更事件

-- 变更事件表（新增）
CREATE TABLE enrollment_event (
  id BIGINT PRIMARY KEY,
  type VARCHAR(20),  -- transfer/promotion/leave/return/dissolution
  student_id BIGINT,
  from_class_id BIGINT,
  to_class_id BIGINT,
  payload JSON,       -- 额外信息
  operated_by BIGINT, -- 操作人
  operated_at DATETIME,
  remark VARCHAR(255)
);

-- 学籍状态（学生表新增字段）
ALTER TABLE student ADD COLUMN enrollment_status VARCHAR(20)
  DEFAULT 'active';  -- active/suspended/transferred/graduated

-- 班级表新增字段
ALTER TABLE class ADD COLUMN status VARCHAR(20)
  DEFAULT 'active';  -- active/dissolved/archived
ALTER TABLE class ADD COLUMN academic_year VARCHAR(10); -- 2025-2026
```

### 关键查询模式

```sql
-- 查询"A 班在 2025 年 3 月的学生名单"
SELECT s.* FROM student s
JOIN student_class sc ON s.id = sc.student_id
WHERE sc.class_id = 'A班ID'
  AND sc.start_date <= '2025-03-31'
  AND (sc.end_date IS NULL OR sc.end_date >= '2025-03-01');

-- 查询"张三的完整班级历史"
SELECT c.name, sc.start_date, sc.end_date, ee.type as change_reason
FROM student_class sc
JOIN class c ON sc.class_id = c.id
LEFT JOIN enrollment_event ee ON sc.event_id = ee.id
WHERE sc.student_id = '张三ID'
ORDER BY sc.start_date;
```
