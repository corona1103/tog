# 换班功能产品方案（v2）

## 一、需求背景

### 1.1 客户反馈与线上问题

| 来源 | 问题描述 | 影响 |
|:-----|:---------|:-----|
| 客户反馈 | 学校管理-学生管理页面「所属班级」功能仅支持查看和删除，缺少直接调整班级的入口；学校实际高频场景是学生转班，当前只能先删除再重新添加，操作繁琐且易出错 | 教务操作效率低，误操作风险高 |
| 线上问题 | 目前「退班再加入新班」的换班方式会导致学生学情数据丢失——例如学生10月在1班，12月换到2班后，该学生10月份在1班的作业数据在任何学情视图中都无法查到 | 老师无法完整了解学生学习情况，数据资产流失 |
| 业务诉求 | 高中选科分班（物化生/物化政/史地生等组合）涉及整个年级数百人重新分班，目前无批量操作能力，教务需逐人手动删除+添加 | 大规模场景完全不可用 |

### 1.2 现状分析

**当前换班的"变通操作"：** 学生管理 > 所属班级 > 删除原班 → 班级管理 > 目标班 > 学生管理 > 新增学生

**该操作的核心问题：**
1. **数据断裂**：`edu_class_student` 关系删除后，原班级学情统计中该学生数据不再可见（因为学情按当前班级的学生列表查询）
2. **无操作记录**：没有换班流水，无法追溯学生的班级变动历史
3. **效率极低**：批量场景（选科分班/学年调整）涉及上百人，逐个操作不可接受
4. **入口割裂**：「退班」在所属班级弹窗操作，「加入新班」需跳转到另一个班级页面，跨页面操作易遗漏

---

## 二、需求目标

| 目标 | 衡量标准 |
|:-----|:---------|
| 提供完整的换班操作能力 | 支持单人换班、单人退班、批量换班三种操作模式 |
| 保障换班后数据不丢失 | 换出学生的历史作业数据在原班仍可查看（标记「已退班」）；新班老师可查看学生历史班级学情 |
| 大幅提升批量操作效率 | 教务上传需调整的学生名单，系统自动解析并执行，无需逐人手动操作 |
| 降低教务的操作门槛和出错率 | <font color="red">模板极简（5列，原班级ID和修改后学号可选填）</font>、未涉及学生不受影响、执行前展示预览供确认 |
| 建立班级变动的可追溯机制 | 所有换班操作写入流水记录，支持审计和回溯 |

## 三、关键决策与思考

### Q1：换班后，原班级的历史学情数据如何处理？

**痛点：** 当前换班（删除+新增）后，学生在原班的作业数据在学情中完全消失，老师无从了解完整学习轨迹。

**决策：**
- 原班级视角：作业任务详情中保留已退班学生的作答数据，姓名后标注「已退班」标签，确保老师能看到当时的完整班级作业情况
- 新班老师视角：学生学情页面默认展示所选时间段内所有班级（含已转出班级）的综合学情，无需手动筛选
- 底层数据：`edu_homework_stu` 表中的历史记录不做任何删除或修改，保持数据完整性

### Q2：批量换班的模板设计，如何降低教务的操作负担？

**痛点：** 教务需要明确指定哪些学生需要调整，同时避免因上传不完整名单导致其他学生被误退班。

**决策：**
- <font color="red">模板 5 列（学生姓名 + 学号 + 原班级ID + 目标班级ID + 修改后学号），教务只需填写需要调整的学生</font>
- 目标班级ID填写规则简单：填班级ID=换班，填-1=退班，与当前班级相同=跳过
- <font color="red">修改后学号为选填，为空则不修改学号；系统自动处理学号冲突检测和链式依赖</font>
- 文件中未涉及的学生完全不受影响，杜绝误操作风险
- 执行前展示变动预览，教务确认后才执行

### Q3：确认的设计前提

1. **"班级"至少是两种东西：行政班（人的归属）和走班（课的归属）。** 系统已有此区分。
2. **学生-班级关系必须带时间维度。** 不是"张三属于 3 班"，而是"张三从 2025-09-01 到 2026-01-15 属于 3 班"。
3. **数据口径问题本质上是产品决策。** "A 班历史统计是否包含已转走的学生"需要明确的默认规则。
4. **"升年级"有两种模式：** A）班级整体升级（班不散人不变）；B）打散重组（旧班消亡新班诞生）。系统需同时支持。
5. 你们在做的是一个**时间维度的学生-组织关系图**。每条关系都是有时间边界的边，任何报表都可以按时间点或时间范围查询（"3 月 15 日那天 3 班有谁？" vs "整个学期贡献了 3 班聚合数据的都有谁？"）它是一个**事件溯源的学籍台账**，数据源是"变更事件的序列"，"当前状态"只是最新的投影（projection）。做对了，就从根本上消灭了"数据孤儿"类 bug，让归属口径变成查询时的参数而非数据迁移问题。

### Q4：学情数据的默认规则（参考建议）

- **班级维度聚合统计（如"A 班 3 月考勤率"）：** 包含当时所有在班学生（含后来转走的），因为他们确实贡献了那个月的数据
- **学生个人维度（如"张三的学情档案"）：** 按时间线完整展示，跨班跨年级无断裂，清晰标注"高一(3)班期间"/"高二(5)班期间"
- **原班升年级的情况：** 通过班级id不变或者存储继承关系，来保障可以查询当前班级在上一学年发生的数据情况
- <span style="color:red;">ps. 此模块的问题，未在本次需求中被完整的处理和解决，需要学情看板相关产品明确需求场景和统计口径</span>

## 四、业务场景梳理

## 六、批量换班功能设计

### 6.1 页面位置

学校管理 > 学生管理 > 「批量导入」Tab 页面内，在现有「批量新增」「批量修改」下方，新增「批量换班」区域。

**页面布局：**
```
模板下载        [下载导入模板]  [下载批量换班模板] ← 新增独立模板
               请根据操作类型选择对应模板

批量新增       [选择文件] [提交]
               仅支持上传xls、xlsx文件，第一行为示例数据，将自动跳过

批量修改       [选择文件] [提交]
               仅支持上传xls、xlsx文件，按原学号匹配并修改学号/密码

批量换班       [选择文件] [提交]                              ← 新增
               仅支持上传xls、xlsx文件，按学生逐条指定目标班级，
               系统自动识别换班/退班操作，文件中未涉及的学生不受影响

─────────────────────────────────────────────────────────────
文件名称 | 导入状态 | 成功/失败记录 | 操作时间 | 操作人 | 操作
```

### 6.2 操作流程

```
下载模板 → 填写需调整的学生及目标班级 → 上传文件 → 系统逐条解析并校验 → 展示变动预览 → 确认执行 → 完成
```

### 6.3 导入模板设计

**文件名：** 批量换班模板.xlsx

**<font color="red">模板 5 列：</font>**

| 列名 | 必填 | 说明 |
|:-----|:-----|:-----|
| 学生姓名 | 是 | 用于人工核对，系统做一致性校验 |
| 学生学号 | 是 | 用于唯一定位学生 |
| 原班级ID | 条件必填 | 指定从哪个班级退出。行政班可不填（系统自动识别）；走班换班时若学生有多个走班则必填 |
| 目标班级ID | 是 | 调整后应归属的班级ID；填写 `-1` 表示退班 |
| <font color="red">修改后学号</font> | <font color="red">否</font> | <font color="red">选填。为空则不修改学号；填写后系统将学号修改为该值（需全局唯一）</font> |

**原班级ID 填写规则：**
- 目标是行政班：可不填（系统自动从当前行政班退出，行政班唯一性约束）
- 目标是走班且学生当前有多个走班：必填（指定从哪个走班退出）
- 目标是走班且学生当前只有1个走班：可不填（系统自动识别）
- 目标为 -1 且不填原班级ID：退出该学生所有班级
- 目标为 -1 且填写了原班级ID：仅退出指定的班级

**模板示例（第一行为说明行，自动跳过）：**

| 学生姓名 | 学生学号 | 原班级ID | 目标班级ID | <font color="red">修改后学号</font> |
|:---------|:---------|:---------|:-----------|:-----------|
| 张三 | G111 | | 1384 | <font color="red">G301</font> |
| 李四 | G222 | 1224 | 1385 | <font color="red">G302</font> |
| 王五 | G333 | | -1 | |
| 赵六 | G444 | 1010 | -1 | <font color="red">G500</font> |

> 示例说明：张三行政班换班并改学号为G301；李四从走班1224换到走班1385并改学号为G302；王五退出所有班级（学号不变）；赵六仅退出班级1010并改学号为G500

**使用说明（模板内 Sheet2 或说明行）：**
> 仅需填写需要调整班级的学生。文件中未涉及的学生不受任何影响。
> 原班级ID：行政班调整时可不填；走班调整时如有多个走班请指定原班级ID。
> 目标班级ID填-1表示退班。不填原班级ID时-1退出所有班级，填了则只退出指定班级。
> <font color="red">修改后学号：选填，为空则不修改。学号在系统内必须唯一，存在冲突时该行将标记失败。</font>

### 6.4 系统逐条解析逻辑

**Step 1：逐行解析，确定原班级**
- 根据学号查询学生当前所有班级
- 确定原班级：若填写了原班级ID则使用填写值；若未填写，按目标班级类型自动匹配（行政班→取当前行政班，走班且仅1个→取该走班）

**Step 2：逐条判定操作类型**

| 目标班级ID | 原班级ID | 判定结果 | 系统动作 |
|:-----------|:---------|:---------|:---------|
| 有效ID，与原班级不同 | 已确定 | 换班 | 从原班退出 + 加入目标班级 |
| 有效ID，与原班级相同 | 已确定 | 跳过 | 不做任何操作 |
| `-1` | 未填写 | 退出所有班级 | 退出该学生当前所有班级 |
| `-1` | 已填写 | 退出指定班级 | 仅退出原班级ID指定的班级 |
| 系统中不存在的ID | — | 异常 | 预览时标红提示"班级ID异常"，执行时跳过 |

**Step 3：行政班唯一性自动处理**
- 目标为行政班且未填原班级ID时，系统自动识别学生当前行政班作为退出对象
- 此行为在预览中明确展示给用户（"原班级"列显示系统自动识别的班级）

**<font color="red">Step 4：学号修改全局冲突检测</font>**

<font color="red">若"修改后学号"列非空，系统进行全局冲突检测：</font>

| 情况 | 检测逻辑 | 处理方式 |
|:-----|:---------|:---------|
| 修改后学号为空 | 不修改学号 | 跳过学号处理 |
| 修改后学号与原学号一致 | 无变化 | 跳过学号处理 |
| 与系统内非本批次学生学号冲突 | <font color="red">目标学号已被其他学生占用，且该学生不在本次文件中</font> | 该行标记失败："学号冲突：已被学生[姓名]占用" |
| 本批次内多人目标学号相同 | <font color="red">文件中两条及以上记录的修改后学号相同</font> | 相关行均标记失败："学号冲突：文件内多人目标学号重复" |
| 链式依赖 |A(G001→G002)，B(G002→G003) — A的目标是B的当前学号，但B也在修改 | <font color="red"> 系统自动拓扑排序确定执行顺序（先B后A），预览中标注执行顺序</font> |
| 环形互换 | A(G001→G002)，B(G002→G001) — 互相占用对方学号 | <font color="red">系统使用临时中间值过渡（先A→临时值，再B→G001，再A→G002），对用户透明 </font>|

### 6.5 校验规则

| 校验项 | 校验逻辑 | 失败提示 |
|:-------|:---------|:---------|
| 目标班级ID有效性 | 目标班级ID在本机构存在且状态正常（-1除外） | 班级ID异常：不存在或已归档 |
| 原班级ID有效性 | 若填写了原班级ID，必须是该学生当前所在的班级 | 原班级ID有误：该学生当前不在此班级中 |
| 走班原班级必填 | 目标为走班且学生当前有多个走班时，原班级ID必须填写 | 该学生有多个走班，请指定原班级ID |
| 学号存在性 | 学号在本机构 `edu_student` 中存在且状态正常 | 不存在该学生，如新生请直接批量导入新增学员即可 |
| 姓名一致性 | 学号对应姓名与填写姓名一致 | 姓名与学号不匹配 |
| 学生重复 | 同一学生（学号+原班级ID组合）不能在文件中出现多次；同一学生不同原班级ID视为不同记录（如同时换行政班+换走班） | 该学生在文件中重复出现 |
| 班级类型一致性 | 换班时目标班级类型必须与原班级类型一致（行政班→行政班，走班→走班） | 班级类型不匹配，不允许交叉换班 |
| 退班有效性 | 填写-1时，学生必须当前在某个班级中 | 该学生当前不在任何班级中，无法退班 |
| 原班级ID与-1组合 | 目标为-1且填写了原班级ID时，原班级ID必须是该学生当前所在的班级（与"原班级ID有效性"一致） | 原班级ID有误：该学生当前不在此班级中 |
| <font color="red">学号唯一性（系统级）</font> | <font color="red">修改后学号不能与系统内非本批次学生的现有学号重复</font> | <font color="red">学号冲突：已被学生[姓名]占用</font> |
| <font color="red">学号唯一性（批次内）</font> | <font color="red">本批次内多条记录的修改后学号不能相同</font> | <font color="red">学号冲突：文件内多人目标学号重复</font> |
| <font color="red">学号格式</font> | <font color="red">修改后学号不为空时，需符合学号格式规则（非空字符串、无特殊字符）</font> | <font color="red">学号格式不合法</font> |

校验不通过的记录执行时跳过，不影响其他正常记录。具体展示见 6.6 变动预览页面。

### 6.6 变动预览页面

上传文件后，系统逐条解析并展示变动预览（弹窗或新页面）：

**顶部汇总：**

### 6.8 执行逻辑（后端）

**调入操作（学生从A班转入B班）：**
1. 从原班级移除：删除 `edu_class_student` 中 A班+该学生的记录
2. 清除原班分组：删除 `edu_class_group_student` 中该学生在A班的分组关系
3. 加入目标班级：新增 `edu_class_student` 记录（B班+该学生）
4. <font color="red">若有学号修改：更新 `edu_student`.student_no 为新学号</font>
5. 写入换班流水：`edu_class_transfer_log`（from_class=A, to_class=B）

**调出操作（学生从A班退出，无目标班级）：**
1. 从原班级移除：删除 `edu_class_student` 中对应关系
2. 清除原班分组：删除 `edu_class_group_student` 中对应关系
3. <font color="red">若有学号修改：更新 `edu_student`.student_no 为新学号</font>
4. 写入换班流水：`edu_class_transfer_log`（from_class=A, to_class=NULL）
5. 不自动禁用账号（退班≠退学，账号禁用由管理员在学生管理中单独操作）

<font color="red">**学号修改执行策略（存在链式/互换依赖时）：**</font>

<font color="red">1. 系统对所有学号变更记录构建依赖图（当前学号→目标学号）</font>
<font color="red">2. 拓扑排序确定执行顺序：无依赖的先执行，被依赖的后执行</font>
<font color="red">3. 环形依赖（如A→B, B→A互换）：引入临时中间值过渡，对用户不可见</font>
<font color="red">4. 同一行的班级调整和学号修改作为一个事务：任一失败则整行回滚，标记失败原因</font>

### 7.3 换班操作交互（选择目标班级）

点击「换班」后，在当前行下方展开内联区域（或弹窗），包含：

| 字段 | 类型 | 说明 |
|:-----|:-----|:-----|
| 原班级 | 只读 | 当前要转出的班级名称及类型 |
| 目标年级 | 下拉选择 | 默认选中与原班级相同的年级 |
| 目标班级 | 级联下拉 | 根据所选年级动态加载该年级下**状态正常（未归档）且与原班级类型一致**的班级列表（行政班只展示行政班，走班只展示走班） |

**交互规则：**
- 目标年级默认选中原班级所在年级，用户可切换其他年级
- 目标班级列表根据年级级联加载，且只展示与原班级**同类型**的班级（行政班换行政班，走班换走班，不允许交叉换班）
- 若用户选择的目标班级与当前原班级相同，回显提示：「与当前班级一致，请重新检查」，确认按钮置灰不可点击
- 目标班级与原班级不一致时，确认换班按钮才可点击
- 确认后立即执行，toast 提示：「换班成功」
- 班级/学生列表刷新

### 7.4 账号禁用联动

- 退班不自动禁用账号（退班≠退学，决策理由见 Q3）
- 禁用时自动退班：管理员在学生列表点击「禁用」时，弹窗提醒"禁用将自动进行学生的退班动作"，确认后系统自动将该学生从所有班级退出
- 自动退班写入 `edu_class_transfer_log` 流水记录
- 学情/作业影响：与正常退班一致（作业详情标注「已退班」，班级学情不再包含该学生未来数据）

## 九、学情数据处理策略

### 9.1 学生学情处理

| 模块 | 当前逻辑 | 换班后表现 |
|:-----|:---------|:-----------|
| （不变）班级学情-任务 | 按当前班级ID查任务 | 换入学生：原班任务不计入新班统计；换出学生：不删除历史任务数据统计 |
| （不变）班级学情-自学 | 按当前班级学生的自学数据汇总 | 换入学生：换班后的自学数据计入新班；换出学生：不删除历史学情数据统计 |
| 学生学情 | 按学生维度查看 | 默认展示所选时间段内所有班级（含已转出）的综合学情，无需手动筛选班级 |
| 作业任务详情 | 按任务+班级查历史提交 | 换出学生的历史提交数据保留在原班任务中 |

### 9.2 作业&靶向任务详情页：已退班学生数据回显

**需求：** 班级维度查看某次作业任务详情时，老师需要看到所有曾参与该任务的学生数据（包括已退班学生），以完整了解当时的班级作业情况。

**方案：** 作业任务详情右侧面板 >「学生详情」Tab 的学生列表中，展示所有提交过该任务的学生（含已退班学生）

**展示规则：**

| 条件 | 展示方式 |
|:-----|:---------|
| 当前仍在本班的学生 | 正常展示姓名 |
| 已从本班转出的学生 | 姓名后标注标签「已退班」，如：张三 `已退班` |

**交互细节：**
- 「已退班」标签样式：灰色背景圆角标签，跟随姓名后方，与姓名间隔8px
- 列表排序：在班学生排在前面，已退班学生排在列表末尾（同一区块内按提交时间排序）
- 已退班学生的作业作答数据正常展示（正确率、提交状态、作答详情等），不做隐藏
- 班级作业统计汇总数据（如全班正确率、提交率）：**仅统计当时在班学生**（即该任务发布时在班的学生），不因后续退班而剔除

**判断规则：** 学生当前是否仍在该班级中。在班→正常展示，不在班→标注「已退班」。退班后又重新加入的学生视为在班。

## 十一、异常与边界case

| 场景 | 处理方式 |
|:-----|:---------|
| 单人换班时目标班级与原班级类型不一致 | 目标班级列表中不展示不同类型的班级（行政班只能换行政班，走班只能换走班），从UI层面杜绝交叉换班 |
| 批量换班时目标班级与学生原班级类型不一致 | 校验阶段拦截，该条记录标记为校验失败，提示"班级类型不匹配"，执行时跳过 |
| 文件中未涉及的学生 | 完全不受影响，不做任何班级调整 |
| 目标班级ID异常（不存在/已归档/-1但无班级） | 校验失败，预览标红提示具体原因，执行时跳过 |
| 目标班级ID与学生当前班级一致 | 视为无需操作，跳过该条数据 |
| 学生同时在行政班A和走班B，文件中指定换到行政班C | 只影响行政班关系（从A换到C），走班B不受影响 |
| 目标班级已满（如有人数上限配置） | 当前版本不限制班级人数，暂不校验；后续版本可扩展 |
| 换班执行中途失败（部分成功） | 每条记录独立执行，互不影响；失败记录标记原因可导出 |
| 换班后学生的班级分组数据 | 自动清除原班分组关系；不自动加入新班分组（需老师手动分组） |
| 换班后未结课的课节 | 学生即刻获取新班的未来课表，不再关联原班课节 |
| 上传文件为空或只有表头 | 提示：文件中无有效数据 |
| 对比后无任何变动 | 提示：当前名单与系统一致，无需调整 |
| 学生当前没有任何班级（调整班级弹窗） | 弹窗内不展示换班/退班按钮，仅展示「加入班级」按钮 |
| 批量换班目标为行政班，但学生当前没有行政班 | 视为首次加入行政班，直接加入目标班级，无需退出操作 |
| 批量文件中同一学生出现多条记录（不同原班级ID） | 允许，按学号+原班级ID去重；如同一学生需同时换行政班和换走班，填两行即可 |
| 已退班学生标签逻辑对历史数据的影响 | 仅对功能上线后发生换班的学生生效；上线前已退出的学生不回溯处理 |
| <font color="red">修改后学号与系统内其他学生冲突（该学生不在本批次中）</font> | <font color="red">该行标记校验失败，提示"学号冲突：已被学生[姓名]占用"，执行时跳过</font> |
| <font color="red">本批次内多条记录修改后学号相同</font> | <font color="red">相关行均标记校验失败，提示"学号冲突：文件内多人目标学号重复"</font> |
| <font color="red">链式学号依赖（A→B, B→C）</font> | <font color="red">系统拓扑排序后按正确顺序执行（先C再B再A），用户无需关心顺序</font> |
| <font color="red">环形学号互换（A→B, B→A）</font> | <font color="red">系统用临时中间值过渡，对用户透明，预览显示为正常通过</font> |
| <font color="red">同一行换班成功但学号修改失败（如并发冲突）</font> | <font color="red">同行作为一个事务，任一失败则整行回滚，标记失败原因</font> |
| <font color="red">仅修改学号不换班（目标班级ID与原班级一致，但填写了修改后学号）</font> | <font color="red">班级操作跳过，仅执行学号修改</font> |
