# PRD：抢答功能

> 版本：v3.0 | 日期：2026-06-17 | 作者：yuj

---

## 一、需求背景

### 现状分析

| 场景 | 问题 |
|------|------|
| 教师讲完知识点想快速验证理解 | 口头提问后只有固定几个学生举手，大部分沉默 |
| 有学生想主动回答但没通道 | 传统举手受物理限制（后排看不到）和心理限制（不好意思） |
| 课堂节奏单一 | 全是"教师发起→学生被动响应"模式，缺乏"学生主动争取"环节 |
| 教师想区分学生主动性 | 当前数据只记录对错，无法区分主动/被动参与 |
| 竞品能力对比 | 希沃课堂已具备抢答功能，当前产品缺乏对应能力 |

---

## 二、需求目标

| 目标 | 衡量标准 |
|------|---------|
| 激发学生主动参与 | 使用抢答时，参与点击学生比例≥60% |
| 均衡参与覆盖 | 每人每课抢到次数方差≤2 |
| 不干扰教学节奏 | 单次抢答从发起到结果展示≤15秒，每课使用≤8次 |
| 与积分体系打通 | 100%抢答行为产生积分变动 |

---

## 三、需求必要性论证与设计思路

### Q1：学生不主动参与，根因是缺工具还是社交压力？

**挑战：** 初中生沉默可能不是"没有通道"，而是不想在同龄人面前暴露自己。工具降低了物理门槛，但社交压力仍在。

**设计应对：**
- 隐藏"是否参与"信息：系统不公示谁点了谁没点，未抢到是多数人的常态，不构成暴露
- 3人同时中签：分散注意力，避免 solo spotlight，降低被盯着的压力
- 按钮始终可点不变色：旁边同学无法通过屏幕判断你是否已参与
- 本质策略：让"参与"变成低风险行为——点了不一定被选中，没点也没人知道

### Q2：使用频次是否足以支撑开发投入？

**挑战：** 抢答天然适合"有标准答案的快速验证"场景，初中课堂中这类时机有限，功能可能用完一周就闲置。

**设计定位：**
- 抢答不是独立功能，是教师组织课堂的多个脚手架之一，与随机点名（被动选人）、随堂检测（全员答题）组合使用
- 预期频次：每个工具平均每周使用1次。对比传统线下课堂（零数字化互动），每周有3-4次不同形式的数字化互动已经是课堂改革的有力证据
- 价值衡量：不以单功能使用频次论成败，以"教师是否建立了使用多工具组合的习惯"为成功标准

### Q3：参与率达标后，如何证明教学价值而非"形式大于内容"？

**挑战：** 学生可能只是条件反射式点击赚积分，"手指在动"不等于"脑子在想"。

**设计回应：**
- 活跃课堂的形式本身具有教学意义：打破沉默→建立互动节奏→学生从"被动接收"进入"准备回应"的状态，这本身就改变了认知姿态
- 不在单功能层面证明认知价值，而是做长期相关性分析：多个工具（抢答+点名+随堂测）的使用综合评定一节课的"学生活跃度指标"，分析活跃度与课后作业正确率的相关性
- 本期只采集数据不下结论，积累一个学期的数据后输出分析报告

## 六、功能设计

### 6.1 抢答主流程

**入口：** 大屏工具箱"抢答" / 教师小屏快捷栏"抢答"

**流程：** 点击发起 → 倒计时3秒 → 抢答中3秒 → 结果展示

**发起规则：**
- 点击即触发，无配置、无确认
- 误触通过"关闭"按钮恢复（见6.5节）

**抢答阶段规则：**

| 阶段 | 时长 | 大屏 | 教师小屏 | 学生端 |
|------|------|------|---------|--------|
| 倒计时 | 3秒 | 弹窗：倒计时数字 | "抢答进行中" | 倒计时数字 |
| 抢答中 | 3秒 | 弹窗：计时器+参与人数 | "抢答进行中" | 大圆"抢"按钮 |
| 结果揭晓 | 持续至操作 | 弹窗：中签名单+操作按钮 | 结果卡片+操作按钮 | 抢到→恭喜 / 未抢到→鼓励 |

**学生端点击交互：**
- 按钮始终保持橙色高亮可点击状态，不变色、不置灰
- 每次点击触发"抢答+1"固定文字从按钮正上方浮出并向上消失（约1秒）
- 多次点击动效独立叠加；仅首次点击计入参与统计

**结果页操作：**

| 按钮 | 行为 |
|------|------|
| 再来一轮 | 直接进入下一轮倒计时 |
| 幸运星 🌟 | 进入幸运星选人流程 |
| 关闭 | 终止抢答，三端回到待机 |

---

### 6.2 后台中签策略引擎

**核心原则：** 学生端体验一致（都能点击），后台决定谁"抢到"。前2名奖励手速，第3名奖励坚持。

**算法流程：**

```
Step 0: 剔除
  - 本课已抢到≥3次的学生移出候选池
  - 豁免：剔除后候选池不足3人 → 不执行剔除
  - 豁免：全班所有人都已≥3次 → 放弃剔除，全员参与

Step 1: 位置1-2（手速槽）
  - 按服务端时间排序，取最快的2人

Step 2: 位置3（补偿槽）
  - 从剩余候选池中筛选：本课参与过点击 且 中签次数=0 的学生
  - 排序：按本课累计点击轮次数倒序（越积极越优先），相同则取本轮更快的
  - 取第1名占位置3
  - 如果无人满足条件 → 取全局第3快的学生
```

**特殊情况：**
- 点击人数 < 3：全部中签，不走上述逻辑
- 点击人数 = 0：无人中签，进入幸运星
- 零参与（本课从未点击）的学生：不进入任何候选池，不强制中签

**对学生透明度：** 完全不透明，学生感知 = "手快更容易抢到"。

---

### 6.3 幸运星功能

**触发方式：**
1. 教师点击"随机幸运星"按钮（结果页/无人抢答页）
2. 无人抢答时系统自动触发

**流程：** 触发 → 姓名滚动（约2.5秒） → 停止选中 → 展示结果+操作按钮

**标题区分：**
- 正常触发："🌟 幸运星降临！🌟"
- 无人抢答自动触发："无人抢答，幸运星降临！"

**选中策略（优先级排序）：**
1. 本课从未参与过点击的学生
2. 参与过点击但从未抢到的学生
3. 抢到次数最少的学生
4. 以上均相同则纯随机

**积分：** 选中即+2分

**结果页操作：** 再来一轮 / 再抽幸运星 / 关闭（同6.1节结果页）

---

### 6.4 积分规则

| 行为 | 积分 |
|------|------|
| 抢答抢到 | +1 |
| 幸运星被选中 | +2 |

结算时机：结果展示时即时结算。

---

### 6.5 全状态关闭能力

**覆盖范围：** 倒计时 / 抢答中 / 结果揭晓 / 无人抢答 / 幸运星 — 所有阶段均可关闭。

**关闭按钮位置：** 大屏弹窗底部居中 + 教师小屏各状态页面底部。

**关闭行为：**
1. 清除所有进行中的定时器
2. 三端同步回到待机（大屏恢复课件、教师小屏回待机、学生端回听课状态）
3. 已产生结果（已中签已结算）→ 积分保留不撤回
4. 未产生结果（倒计时/抢答中途）→ 本轮作废不记录

---

### 6.6 节奏提示

连续发起4轮抢答后，结果页底部显示：`💡 已连续4轮抢答，试试"幸运星"让更多同学参与？`

- 仅文字，不弹窗
- 每课最多提示1次

## 七、异常与边界Case

### 7.1 发起与关闭

| 场景 | 处理方式 |
|------|---------|
| 教师误触发起 | 全阶段"关闭"可用，点击后三端回到待机 |
| 倒计时阶段关闭 | 本轮作废，不产生任何记录 |
| 抢答中途关闭 | 本轮作废，已有点击不计入结果 |
| 结果展示后关闭 | 已产生积分保留，仅关闭弹窗 |
| 幸运星滚动中关闭（未停止） | 本次幸运星作废不记录 |
| 幸运星已选出后关闭 | 积分保留，关闭弹窗 |
| 大屏和教师小屏同时操作 | 以服务端收到的最后一条指令为准，广播所有端 |

### 7.2 抢答参与

| 场景 | 处理方式 |
|------|---------|
| 无人点击（3秒超时） | 跳过过渡，直接进入幸运星（标题："无人抢答，幸运星降临！"） |
| 点击人数 < 3 | 所有点击学生全部中签（不受次数限制） |
| 毫秒级并发点击 | 以服务端时间戳为准，精确到毫秒 |
| 网络延迟致"先点后到" | 以服务端时间为准（已知局限） |
| 全班都已抢到3次 | 豁免剔除规则，按纯时间排序取前3 |
| 学生连续快速点击 | 每次触发浮动动效；仅首次计入参与统计 |

### 7.3 设备与网络

| 场景 | 处理方式 |
|------|---------|
| 学生设备离线 | 不显示抢答按钮；不影响其他学生 |
| 抢答期间学生退出 | 已在候选池则移除，递补下一位 |
| 幸运星选中学生离线 | 重新随机选中另一位，动效重新播放 |
| 教师小屏断线重连 | 同步当前抢答状态，恢复对应页面 |
| 大屏断线重连 | 同步当前状态，恢复对应弹窗 |

### 7.4 流程衔接

| 场景 | 处理方式 |
|------|---------|
| 连续快速发起多轮 | 动效叠加不阻塞，新轮立即开始 |
| 幸运星全班都已被选过 | 降级为纯随机 |
| 本课使用超过8次 | 不强制限制，仅在"再来一轮"旁显示"本课已使用8次" |
| 无人抢答后点"再来一轮" | 大屏从幸运星切换到新一轮倒计时 |

## 附：内部评审意见

### 产品评审

**总体评价：** 功能定义清晰，操作路径简洁，边界Case覆盖较完整。以下几点需补充或调整：

| # | 问题 | 建议 |
|---|------|------|
| P1 | 目标"参与率≥60%"缺乏基线对比 | 补充当前课堂互动（如随堂测）的平均参与率作为基线，否则60%无法判断好坏 |
| P2 | "抢到即得分"对教学价值的传递偏弱 | 建议结果页增加教师"点评标记"能力（轻量，非判对错），如标记"回答精彩"，为下期对错判定做铺垫 |
| P3 | 幸运星优先选"未参与"的学生，可能选中正在走神/离线边缘的学生 | 幸运星候选池应排除"最近5分钟无任何操作"的学生，避免尴尬 |
| P4 | 无人抢答→自动幸运星，教师无选择权 | 建议改为：无人抢答后教师小屏提供"幸运星"和"关闭"两个选项，不强制自动触发，把决策权交给教师 |
| P5 | 缺乏功能灰度/开关策略 | 补充：首次上线时以班级维度灰度，教师可在设置中关闭抢答功能 |
| P6 | 中签策略"插队1-2个位置"表述模糊 | 需明确：权重×1.5 = 向前移动1位，权重×2 = 向前移动2位，还是概率性插队？研发无法实现模糊描述 |

### 技术评审

**总体评价：** 方案可行，但有几个技术风险和实现细节需要对齐：

| # | 问题 | 建议 |
|---|------|------|
| T1 | 3秒抢答窗口 + 服务端时间戳排序 → 对网络延迟敏感 | 建议：客户端记录本地点击时间戳一起上报，服务端取 `min(客户端时间, 服务端收到时间)` 作为排序依据，降低网络抖动影响 |
| T2 | "仅首次点击计入参与统计"但按钮不变色 → 客户端会持续发请求 | 需明确：客户端首次点击后本地标记，后续点击仅触发动效不发网络请求，否则45人×多次点击会造成瞬时并发压力 |
| T3 | 中签策略的"权重加成"在时间排序中如何实现？ | 建议产品明确二选一：A）纯时间排序+末位替换（确定性）；B）加权随机（概率性）。当前描述混合了两种模式，无法实现 |
| T4 | 三端状态同步依赖 WebSocket | 需确认：当前系统是否已有长连接通道？如果是轮询架构，3秒倒计时精度无法保证，需要先评估基础设施改造成本 |
| T5 | "关闭"操作的一致性保证 | 极端情况：教师点关闭的同时，服务端刚好完成中签计算并广播结果。需明确优先级：关闭指令是否可以"撤回"已广播的结果？建议：关闭指令优先，客户端收到关闭后丢弃同时收到的结果消息 |
| T6 | 幸运星"重新选人"的触发条件 | "选中学生离线→重新选"，如何判断离线？心跳超时阈值建议明确（如5秒无心跳视为离线），避免选中后学生短暂网络波动就被替换 |
| T7 | 数据量评估 | 按每课8轮×每轮45人点击 = 360条/课，峰值并发45QPS/班。如果多班同时使用需评估是否需要消息队列削峰 |

### 评审结论

**产品侧：** 条件通过，P4（无人抢答自动触发改为教师选择）和 P6（权重策略明确化）需在开发前修订。

**技术侧：** 条件通过，T3（中签算法明确）和 T4（长连接基础设施确认）为开发前置阻塞项，需产品+技术对齐后方可排期。
